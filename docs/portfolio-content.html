Tier 10In MaGMML, contest entries are sorted into different tiers based on their placement. For MaGMML3, I was given the opportunity to design/code Tier 10.

(Preview of Tier 10 from results stream below)



Tier 10, the Rave Concert, is unique in that almost every element within the Tier is timed to the background music. This is done by interacting with the FMOD DLL to get the timing of the track. Based on the time received from FMOD, the game advances the logic of each object when the time has passed a certain point.

The exact music-timings to use were found by creating a lua script for FCEUX (an NES emulator), designed to read the exact frame at which a note on the NES's 2A03 channels played. These timings are then inserted into MaGMML3's playback code.
Special WheelIn previous MaGMMLs, there was a key titled "Special" bound to what would be the Select button on an NES controller. In all MaGMMLs except Episode Zero, this key would open the map when the Mapper item was obtained. In Episode Zero, this key is instead held in combination with the weapon select keys to switch chips instead of weapon types.

MaGMML3 brings the idea of both of these functions together + more with the introduction of the Special Wheel. By holding down the Special key, you instead freeze gameplay to open up a wheel, from which you can hold certain directions to perform certain actions.

Actions include opening the map, swapping between normal weapons and Alter weapons, using E/W/M-Tanks, as well as other effects.

The player can also choose to bind a specific item in the pause menu to Special, rather than use the wheel, to use it quickly without freezing gameplay. This is primarily intended for speedrunning/leaderboards, who would be penalized for switching to Alter weapons otherwise.
Automap FixesCompared to previous MaGMMLs, a larger number of entries than usual created levels with unusual section border placement. This results in the game's Mapper item recording inaccurate room layouts (or crashing in some rare cases).

In MaG48HMML, we adjusted all levels that would crash, but in the two cases (later one in an update) that were still problematic, we disabled the mapper throughout the level. With MaGMML3, I aimed to fix that.

The solution involves creating a 'fake' level with approximate borders that are still accurate to the level, but adjusted in such a way that while not fully playable, portrays the original level accurately. This output is then "recorded" in-text as it'd be inserted into the mapper as code. That code is then placed into the mapper for the specific levels, with those levels turning off their map generation.
Border SupportA new addition to MaGMML3 is the addition of borders. Primarily intended for fullscreen mode (but usable in windowed mode as well), borders are designed to fill the empty space brought by Mega Man games being in an 8:7 aspect ratio.

Alongside the addition of borders, a special border was programmed specifically for the MaGMML3 Results stream. This variant instead displays the level name and author, as well as score information about the level currently being showcased.

Quick CompilerThe Quick Compiler is a new tool created for MaGMML3, as an evolution to the Lightweight/Featherweight builders of previous entries.

Though the Featherweight on its own was massively helpful the past two projects, MaGMML3 further accentuates other problems held by our GML version, primarily as a result of having over 180 entry levels alone. To solve these problems, I created the GMSQuickCompiler, which addresses several issues:
  • There is no option to retry a previously compiled build, whether you ended the game manually or it crashed, taking even more time to recompile.
    • Fixed by providing a small .bat file with every project that relaunches the game using the user's selected Runner executable. The UI allows the user to immediately select the last-used .bat file.
  • Temp file management is poor, and requires manual deletion if you choose to End Task the IDE.
    • Fixed by creating a temp file manager in the UI that allows you to delete all temp files at once.
  • Rooms require opening the entire project to edit most of the time (takes upwards of 30 minutes on low-end PC's)
    • Fixed by creating an entirely new Room Editor that relies on grabbing assets only used by the room, until the user asks to add assets.

Texture Page OptimizerIn the GameMaker engine, sprites are seperated into seperate texture pages, organized by groups of pages. The default option compiles them all into a single texture group, which takes an unacceptable amount of time to compile for MaGMML games, as the compiler tries to figure out the best way to fit sprites on each page.

In previous MaGMMLs, we would use a texture page randomizer to randomly assign each sprite in the game prior to its release. This would solve compiler times as well as better disperse sprites across different texture groups.

When running the game however, any time a sprite needs to be drawn, the currently running texture page needs to be swapped out if the next sprite is on a different page. With enough swapping, this can result in lag on lower-end PC's.

The Texture Page Optimizer was made to address this: Rather than randomly assign each sprite to a different group, groups are created based on their location in the project file's hierarchy.

Under this algorithm, devkit assets are all placed in groups based on their purposes (Enemies have their own group, Gimmicks have their own group, etc.).

Entry assets, on the other hand, are grouped based on their author. This allows the entries themselves to have minimal texture page swapping, as those that use custom assets all-around will only need to perform a few swaps, while those that mix custom and devkit assets only swap between two pages, as opposed to multiple randomized pages.

Finally, certain special groups are assigned for very large groups, in order to ensure they only have a single texture page per-sprite. Currently, these groups include costume spritesheets and border UI.

2022

ARClassic

Project Leader/Programmer ARClassic is a mod for Super Smash Bros. Ultimate that expands upon the Classic Mode of the game. In normal gameplay, players can only choose one specific Classic Mode Route for every fighter, dependant on the fighter chosen. This makes the variety in the mode itself stale if the user wishes to stick to a particular character.

ARClassic changes this by allowing you to select between every fighter's route, no matter which fighter you choose. It also allows you to add custom routes without directly replacing a particular character's route.

WonderCorner.net

Project Leader/Programmer/Webmaster WonderCorner is an online interface designed for mainline Nintendo DS Pokemon games. It is an alteration of the protocol used for the Global Trade System (GTS) of each game, in order to have it act similar to the Wonder Trade feature introduced in the Nintendo 3DS Pokemon games (with some alterations to account for a smaller userbase).

While the initial base of the project was centered around open-sourced code from 2015, it quickly evolved into a set of classes, subprotocols, and webpages designed to make the user experience as seamless as possible, given the means of execution.

Additional features beyond the DS-oriented protocols include:
  • A web interface showing all Pokemon available for certain 'channels' of trade, as well as receipts for recently traded Pokemon.
  • HTTPS support for the above website.
  • A specialized display system for country flags of each Pokemon for the in-game GeoNET system.
  • Support for displaying Shiny Pokemon that would become "Square Shinies" when transferred to Nintendo Switch Pokemon games.
  • User account options via the DS's interface (Turning on and off location data, as well as GDPR support).
  • Partial DDOS protection.
  • A Discord bot capable of:
    • Displaying every Pokemon trade made by the system.
    • Sending internal statistics (IV's/EV's) of a user's Pokemon by syncing their game with their Discord account.
  • An auto-tweet bot that showcases a chosen set of Pokemon that have been traded daily.
  • Implementation of a legality-check system to ensure Pokemon do not have cheated values when under the "Vanilla" subdomain, as well as a strong safety-check system used for all trades.
  • A system for auto-generating competitively-legal, statistically-accurate Pokemon, used in the event the server runs out of Pokemon in the system, and in the Lucky Channel.

QA TeamEpisode Zero is unlike other MaGMML Contests in the sense that it's not actually a contest! Rather than having entrants send a level to be judged, levels would instead by reviewed by a Quality Assurance team consisting of five devteam members. If the level did not receive an accept vote from at least 3 of 5 judges, it would be rejected. This encouraged a higher quality bar than the main contests tend to provide for entrant levels, as it would encourage entrants to resubmit levels with improvements for a chance at acceptance.

Featherweight Project BuilderAs the MaGMML series is based off of GML, several roadblocks come into play. In particular, compile times become a long timesink as a result of the many assets spread across multiple level entries.

In order to develop the game at a faster pace, a Lightweight Project Builder was created before I was hired that finds any and all assets not used in the current project, and produces a "lightweight" project without those assets.

This was primarily designed for the base-engine in order to create the simplified "Light-Megamix" engine, which includes only some basic enemies and required assets. As a result, it has a problem with full-scale MaGMML projects, as the vast majority of enemies/gimmicks are already used across the entry levels.

My solution was to create a "Featherweight" Project Builder for the purposes of development. In addition to removing all unused assets, the builder asks the user to specify what rooms they'd like to compile with. The builder then takes these rooms (as well as some "core" rooms essential to the project as a whole) and keeps assets only used within that set, reducing the project file (and as a result of compile times) drastically.

2018

Mega Man 2 & Bass

Project Leader/Programmer Mega Man 2 & Bass is a mod for the Mega Man Legacy Collection. It utilizes a combination of Lua script and FCEUX (an NES emulator) to extract ROM contents straight from the Legacy Collection, then runs the game with a sequence of commands on-top that take control of the game's RAM (and in some cases, ROM), to add the character Bass from later Mega Man titles into Mega Man 2.

It also provides support for expansion audio through a special application, allowing the user to play new music in place of the original soundtrack (such as remixes of said soundtrack).
